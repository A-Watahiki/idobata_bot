# 井戸端かいぎ 自動化Bot

「井戸端かいぎの予定表」(Notion)と Discord・Googleカレンダー・Google Drive を
連携させ、承認〜告知〜リマインダーまでを自動化するスクリプト集です。
Make.comなどの外部ノーコードツールを使わず、GitHub Actionsだけで完結します。

会場(Zoomなど)は主催者各自が用意し、申込み時にそのURLを入力してもらいます。
このボット自体はZoomミーティングを自動作成しません。Notionデータベースは
対外公開する予定のため、会場URLはプロパティとして一切持たない設計にしています。
会場URLは Googleカレンダーの予定と、開催直前のDiscordリマインダーにのみ
記載されます。申込者とのやり取りはDiscordのお知らせスレッド上で行います。

Notionは非公開の「井戸端かいぎ 申込み」データベース(フォーム・メールアドレスを
含む)と、公開の「井戸端かいぎの予定表」データベースの2つに分かれています。
申込みはメールアドレスを除いた内容だけが自動的に予定表へ転記されるため、
運営が予定表側の編集権限を主催者へ付与しても、他の主催者のメールアドレスが
見える心配はありません(予定表にはメールアドレスというプロパティ自体が
存在しないため)。

承認後、運営は申込者(主催者)を「井戸端かいぎ 申込み」データベース側の
メールアドレスをもとに、**公開の予定表データベースへ**ゲストとして手動で
招待し、編集権限を付与する運用です(Notion APIには権限を自動付与する機能が
ないため)。編集権限を得た主催者は、日時変更や、複数回シリーズの追加開催
(前回ページの複製)を自分で行えます。

公表前の研究成果を扱うなど、参加者を限定したいイベントは「申込み必須」に
チェックを入れることができます。この場合、会場URLはDiscordの公開チャンネル・
スレッドに一切表示されず、代わりに非公開の「井戸端かいぎ 参加申込み」
データベース(氏名・メールアドレス必須)経由で、事前申込みした人にだけ
会場URLがメールで案内されます。

## 全体の流れ

Notionの「Send webhook」アクションは有料プラン限定のため、GitHub Actionsの
`schedule`(cron)による**定期ポーリング方式**を採用しています。

1. `.github/workflows/poll.yml` が10分おきに自動実行され、
   - `sync_submissions.py`: 非公開の「井戸端かいぎ 申込み」データベースを見て、
     まだ「転記済み」でない申込みを検知し、メールアドレスを除いた内容を
     公開の「井戸端かいぎの予定表」に新しい行として作成(「ステータス」は
     「募集中」)。作成済みの申込みには「転記済み」チェックを立て、
     運営用チャンネルに新規申込みの通知(承認手順の案内つき)を投稿する
   - `poll_approve.py`: 「ステータス」が「確定」なのに「Discordスレッド」が
     未作成の行を検知し、
     - 会場URLを特定(「シリーズ名」が入力されていて同じシリーズ名の既存
       カレンダー予定があれば、そこから会場URLを再利用。なければ非公開の
       「井戸端かいぎ 申込み」ページに主催者が入力した「会場URL」を読み出す。
       まだ届いていない場合はTODO案内で主催者に用意を促す)
     - Discordにお知らせスレッドを作成(種別・日時・主催者・会場URLなどを記載)
     - Googleカレンダーに予定を作成(スレッドと同じ内容+会場URLを記載。
       Notionには一切書き込まない)
     - スレッドにTODO案内を投稿(「主催者ユーザ名」をメンションしてお知らせ)。
       Googleカレンダーはメンバー向けの非公開の日程確認手段のため、Discordには
       URLを送らない
     - Notionにスレッドリンクを書き戻し
   - `remind_events.py`: Googleカレンダーの予定を見て、「#🐸｜井戸端かいぎ」
     チャンネル全体に(該当スレッドへのリンクつきで)リマインダーを投稿
     - **開催24時間前**: 会場URLなし
     - **開催30分前**: 「申込み必須」でなければ会場URLつき。「申込み必須」の
       場合は会場URLを出さず、代わりに非公開の「井戸端かいぎ 参加申込み」
       データベースから事前申込み者を検索し、一人ひとりに会場URLを再度メールで案内
   - `rsvp_notify.py`: 「井戸端かいぎ 参加申込み」データベースを見て、まだ
     「案内送信済み」でない申込みを検知し、対応するイベントが承認済み
     (Googleカレンダー登録済み)であれば、その会場URLを申込み者へメールで案内
     (「案内送信済み」を立てる)。イベントがまだ未承認の場合は次回以降に再試行
   - `sync_cancellations.py`: Googleカレンダーの今後の予定のうち、対応する
     Notionページが削除・ゴミ箱行き・「ステータス」が「キャンセル」に
     なっているものを検知し、そのカレンダー予定を自動削除
   - `sync_updates.py`: 承認済みイベントのNotionページが編集されていないか
     (last_edited_timeの変化で)検知し、
     - **日時が変更された場合**: Googleカレンダーの時刻を更新し、スレッドと
       「#🐸｜井戸端かいぎ」チャンネル全体(スレッドへのリンクつき)に通知
       (会場URLは主催者自身が管理するため変更しない)
     - **日時以外が変更された場合**: Googleカレンダーの内容を更新し、
       スレッドにのみ通知
   - (Googleカレンダーはこれらのスクリプト以外からは編集しない運用)
2. `.github/workflows/daily_check.yml` が毎日自動実行され、`daily_check.py` が
   開催2日前で資料未共有の発表者に、Discordのお知らせスレッドへリマインダーを投稿

開催時間は基本30分・最大40分を想定しています(議論が延びる場合は一度退出し、
同じZoom会場に入り直す運用)。

## 事前準備

### 1. このリポジトリをGitHubにpush

```bash
cd idobata-kaigi-bot
git init
git add .
git commit -m "init"
git remote add origin https://github.com/A-Watahiki/idobata_bot.git
git push -u origin main
```

このリポジトリは秘密情報を含まないので、Public / Private どちらでも構いません
(Publicの場合、GitHub Actionsの実行時間は無制限・無料です)。

### 2. Googleカレンダーのリフレッシュトークン発行(一度だけ手動)

`doubutsurinrikaigi@gmail.com` でログインした状態で:

1. Google Cloud Console (https://console.cloud.google.com/) で新規プロジェクトを作成(または既存のものを利用)
2. 「APIとサービス」→「ライブラリ」で「Google Calendar API」を有効化
3. 「APIとサービス」→「認証情報」→「OAuthクライアントID」を作成
   - アプリケーションの種類: 「デスクトップアプリ」
   - 発行される `クライアントID` / `クライアントシークレット` を控えておく
4. OAuth同意画面で自分(doubutsurinrikaigi@gmail.com)をテストユーザーとして追加
5. ローカル環境などで以下のようなスクリプトを一度だけ実行し、認可コードからリフレッシュトークンを取得する
   (`google-auth-oauthlib` を使うと簡単です。`pip install google-auth-oauthlib` の上で):
   ```python
   from google_auth_oauthlib.flow import InstalledAppFlow

   flow = InstalledAppFlow.from_client_config(
       {
           "installed": {
               "client_id": "<クライアントID>",
               "client_secret": "<クライアントシークレット>",
               "auth_uri": "https://accounts.google.com/o/oauth2/auth",
               "token_uri": "https://oauth2.googleapis.com/token",
               "redirect_uris": ["http://localhost"],
           }
       },
       scopes=["https://www.googleapis.com/auth/calendar"],
   )
   creds = flow.run_local_server(port=0)
   print("refresh_token:", creds.refresh_token)
   ```
   ブラウザが開くので `doubutsurinrikaigi@gmail.com` でログイン・許可し、
   表示された `refresh_token` を控えておく

### 3. Gmail アプリパスワードの発行(「申込み必須」イベントを使う場合のみ)

「申込み必須」イベントの事前申込み者へ会場URLをメール案内するために使います。
「申込み必須」機能を今すぐ使う予定がなければスキップしても構いません
(未登録のSecretsは空文字列として渡されるだけで、`remind_events.py` /
`rsvp_notify.py` 自体はエラーになりません。「申込み必須」イベントを一度も
作らなければメール送信処理は実行されないため、実害はありません)。

`doubutsurinrikaigi@gmail.com` で:

1. Googleアカウントの「2段階認証プロセス」を有効にする
   (https://myaccount.google.com/security)
2. 「アプリ パスワード」(https://myaccount.google.com/apppasswords) で
   新しいアプリパスワードを発行する(アプリ名は任意。例:「井戸端かいぎBot」)
3. 発行された16桁のパスワードを控えておく(通常のGoogleログインパスワードとは別物)

### 5. Notion側の設定

データベースを**非公開の「井戸端かいぎ 申込み」**と**公開の「井戸端かいぎの
予定表」**の2つに分けて作成してください。フォームは申込み側にだけ設置します。

#### 「井戸端かいぎ 申込み」(非公開・フォームの送信先)

以下のプロパティが必要です(名称は完全一致させてください):
タイトル / 日時 / 種別 / 主催者ユーザ名(テキスト) / メールアドレス(テキストまたはEmail型) /
会場URL(URL型) / 概要 / 対象 / シリーズ名(テキスト) / 申込み必須(チェックボックス) /
転記済み(チェックボックス)

フォームの入力項目は最小限(イベントタイトル・主催者ユーザ名・メールアドレス・
開催日時・種別・対象・概要・シリーズ名・会場URL・申込み必須)にとどめ、
それ以外の詳細は概要欄に書いてもらう運用にしてください。

「申込み必須」は、公表前の研究成果を扱うなど、参加者を限定したいイベントで
主催者にチェックしてもらうためのものです(詳細は後述の「井戸端かいぎ 参加申込み」
の項を参照)。

「会場URL」には、主催者自身が用意したZoomなどのミーティングURLを入力して
もらいます。このボットはZoomミーティングを自動作成しません。**どうしても
自分で会場を用意できない場合は空欄のまま申込んでもらい、承認後にスレッドで
運営へ相談するよう案内します**(TODO案内メッセージに自動で記載されます)。
複数回シリーズの場合、2回目以降はGoogleカレンダー側から自動的に同じ
会場URLが再利用されるため、毎回入力し直す必要はありません。

このデータベースにはメールアドレスが含まれるため、**運営(管理者)以外には
共有しないでください**。「転記済み」チェックは`sync_submissions.py`が自動で
つけるので、手動で操作する必要はありません。

#### 「井戸端かいぎの予定表」(公開)

以下のプロパティが必要です(名称は完全一致させてください):
タイトル / 日時 / 種別 / 主催者ユーザ名(テキスト) /
ステータス / 概要 / 対象 / 資料リンク / 録画 / 議事メモ /
Discordスレッド(URL型) / シリーズ名(テキスト。単発の場合は空欄) /
申込みページID(テキスト) / 申込み必須(チェックボックス)

このデータベースにはメールアドレスや会場URLなど個人情報・非公開情報の
プロパティを一切作らないでください(`sync_submissions.py`もそれらを
転記しません)。行ページを直接開くとその行の全プロパティが見えますが、
そもそもそうした情報を持たない設計なので安全です。

「申込みページID」は、`sync_submissions.py`が転記時に自動で書き込む、
元の申込みページへの内部参照用ID(単なる文字列で、それ自体は個人情報では
ありません)です。承認時に`poll_approve.py`がこのIDを使って申込みページから
会場URLだけを読み出します。運用上は編集不要のため、対外公開ビューでは
非表示にしておくとよいでしょう。

「主催者ユーザ名」はDiscordのユーザー名(`@`から始まる/始まらないどちらでも可)です。
承認時にこれをもとにDiscord上のユーザーを検索し、TODO案内メッセージで直接
メンションします(見つからない場合はプレーンテキストのまま)。

新規申込みがあると、`sync_submissions.py`が運営用チャンネル
(`DISCORD_ADMIN_CHANNEL_ID`)に、タイトル・主催者・日時・種別と、
「申込み」ページ・「予定表」ページそれぞれへのリンクつきで通知します
(メールアドレスそのものはDiscordには書かず、非公開の「申込み」ページへの
リンクのみを示します)。

運営は、この通知またはNotion上で「ステータス」が「募集中」で自動作成
された行を確認し、問題なければ「井戸端かいぎ 申込み」側の該当行にある
メールアドレスを控えて、**予定表データベースの該当行**を開き「共有」メニュー
からそのメールアドレス宛にゲスト招待・編集権限を付与した上で、「ステータス」
を「確定」に変更してください(承認操作)。

またNotionの共有・Web公開設定は**ページ単位**であり、ビュー単位ではありません。
1つのページに複数ビュー(テーブルビュー・カレンダービューなど)をまとめている
場合、ページ全体を「Webに公開」すると、URLの`?v=`部分を書き換えるだけで
他のビューにも到達できてしまいます。予定表データベースは公開前提なので
問題ありませんが、申込みデータベース(非公開)を誤って同じページに置いたり
Web公開したりしないよう注意してください。

#### 「井戸端かいぎ 参加申込み」(非公開・「申込み必須」イベント用)

「申込み必須」イベントに参加したい人が、氏名・メールアドレスを入力して
事前申込みするための、**全イベント共通の1つの**データベースです。
イベントごとに新しいフォームを作る必要はありません。以下のプロパティが
必要です(名称は完全一致させてください):

氏名(タイトル型) / メールアドレス(テキスト) / 参加イベント(リレーション。
「井戸端かいぎの予定表」データベースへの関連付け。1件のみ選択できるよう
「1ページに制限」を有効にしてください) / 案内送信済み(チェックボックス) /
リマインダー送信済み(チェックボックス)

フォームには氏名・メールアドレス・参加イベント(リレーション。プルダウンから
対象の予定表ページを選択)の3項目だけを設置してください。「案内送信済み」
「リマインダー送信済み」は`rsvp_notify.py`/`remind_events.py`が自動で
つけるので、フォームには含めないでください。

フォームを作成したら、そのURLを`scripts/discord_utils.py`冒頭の
`RSVP_FORM_URL`定数に設定してください。「申込み必須」イベントの告知embedに
このURLが自動的に案内文とともに掲載されます。

このデータベースにはメールアドレス・氏名が含まれるため、**運営(管理者)以外には
共有しないでください**。参加者からの申込みを検知すると、`rsvp_notify.py`が
対応するイベントの承認状況(Googleカレンダー登録済みかどうか)を確認し、
承認済みであれば会場URLと機密保持の注意文をメールで案内します(未承認の間は
次回以降のポーリングで再試行し続けます)。さらに開催30分前には、
`remind_events.py`が同じ会場URLを含むリマインダーメールを申込み者全員へ
再送します。会場URLはこのデータベースの外(Discordの公開チャンネル・
スレッドや公開の予定表)には一切表示されません。

#### 複数回シリーズ(輪読会など)の運用

シリーズDBやリレーションは使いません。承認後にNotionの編集権限を得た主催者が、
**前回のイベントページをそのまま複製**(Notionの「•••」メニュー→「複製」、
または「複製」ボタンプロパティ)し、「日時」だけを新しい回の日時に書き換えて
「ステータス」を「確定」に戻すだけで、次の回の告知・カレンダー登録が
自動的に走ります。複製すればタイトル・種別・概要・対象・「シリーズ名」は
そのまま引き継がれるため、シリーズ名を毎回手入力する必要はなく、会場URLも
自動的に(初回申込み時に入力されたものと同じ)ものがGoogleカレンダー経由で
再利用されます(複製すると「申込みページID」も前回の値のまま複製されますが、
`poll_approve.py`はまずシリーズ名でカレンダーからの再利用を試みるため、
この古いIDが実際に参照されることはありません)。

#### 承認後の編集権限付与

上記のとおり、「ステータス」を「確定」に変更する際に、予定表データベースの
該当ページへゲスト招待・編集権限の付与もあわせて行ってください
(この手順はNotion APIでは自動化できません)。権限を得た主催者は、その後の
日時変更・内容修正・複製による追加開催が自分自身でできるようになります。

#### 承認済みイベントの変更検知

編集権限を得た主催者がページを修正すると、`sync_updates.py`が10分以内に
検知します。

- **日時を変更した場合**: Googleカレンダーの時刻が自動更新され、
  スレッドと「#🐸｜井戸端かいぎ」チャンネル全体に通知が届きます
  (会場URLは主催者自身が管理するため変更されません)
- **日時以外を変更した場合**: Googleカレンダーの内容が更新され、
  スレッドにのみ通知が届きます

「ステータス」を「キャンセル」に変更する、またはページ自体を削除(ゴミ箱に移動)
すると、対応するGoogleカレンダーの予定が自動的に削除されます
(`sync_cancellations.py`)。

#### 承認ボタンの設定

アクションは1つだけで構いません:「プロパティを編集」→「ステータス」を「確定」に変更。
(Webhookは使わないため、Webhook送信アクションの設定は不要です)

`poll.yml` が10分以内にこの変更を検知し、カレンダー作成・Discordへの告知・
TODO投稿・スレッドリンクの書き戻しを自動で行います。

## 動作確認

まず `workflow_dispatch` で `poll.yml` と `daily_check.yml` をそれぞれ手動実行して、
Secretsが正しく設定されているか確認するのがおすすめです。
GitHubリポジトリの Actions タブ → 該当ワークフロー → "Run workflow" から実行できます。

## 制限・注意点

- `discord_utils.py` のスレッド作成は、Botに該当チャンネルへのアクセス権限が必要です
- `drive_utils.py` のファイル検索はタイトルの部分一致による簡易的なものです。ファイル名に発表タイトルの一部を含めてアップロードしてもらうよう、事前に周知してください
- 会場(Zoomなど)は主催者各自が用意する運用です。多くの主催者はZoom無料プランを
  使うと想定されるため、「基本30分・最大40分」という開催時間の運用方針は
  引き続き周知してください(Zoom無料プランはグループミーティング(3名以上)が
  40分で自動終了します)
- 申込み時に会場URLが未入力だった場合、承認後のTODO案内で主催者に用意を促し、
  それでも用意できない場合は運営(`@xenamanex`)へ相談するよう案内します。
  会場URLが未設定のまま開催当日を迎えた場合、開催30分前のリマインダーにも
  会場URLは記載されません
- `remind_events.py` のリマインダー、複数回シリーズの会場URL再利用、
  `sync_cancellations.py` のキャンセル同期、`sync_updates.py` の変更検知は、
  いずれもGoogleカレンダーの予定作成時に保存した `extendedProperties`
  (NotionページID・DiscordスレッドID/URL・会場URL・シリーズ名・
  Notionの最終更新時刻)を参照します。そのためGoogleカレンダーの
  予定を手動で複製・作成した場合、これらの機能は正しく動作しません
- `discord_utils.resolve_mention()` はサーバーのメンバー検索APIを使うため、
  Botがそのサーバーに参加しており、対象ユーザーもサーバーメンバーである必要があります
- 「申込み必須」イベントのメール送信は`smtplib`によるGmail SMTP経由の素朴な実装です。
  大量送信や配信停止管理などは想定していません。送信エラー(誤入力のメールアドレス等)は
  そのポーリング回では例外で止まってしまうため、他の申込み者への送信も含めて
  そのポーリング回全体が失敗します(次回のポーリングで、送信済みの人には
  「案内送信済み」フラグにより再送されず、未送信の人にだけ再試行されます)
- `discord_utils.py`の`RSVP_FORM_URL`は「井戸端かいぎ 参加申込み」フォームを
  作成してから、プレースホルダーを実際のURLに書き換える必要があります
  (書き換え忘れると「申込み必須」イベントの告知に無効なURLが載ります)
- 本コードはひな形です。実際の運用前に、テスト用のNotionページ・Discordチャンネル・
  Zoomミーティング・Googleカレンダー・(「申込み必須」を使う場合は)テスト用の
  メールアドレスで一通り動作確認することを強く推奨します
